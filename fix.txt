1. 근거 손실 및 컨텍스트 파편화 (Context Fragmentation)
   * 현상: Focused Gather 단계에서 백엔드(pty-manager.ts)와 프론트엔드(app.ts) 양쪽의 증거를 모두 수집했음에도, Concrete Plan 단계로 넘어오면서 백엔드 수정 계획이 증발하는 현상이 반복됩니다.
   * 결함: '근거 요약(Evidence Consolidation)' 과정에서의 정보 휘발입니다. 여러 개의 증거를 하나로 압축할 때, 모델이 '가장 그럴듯해 보이는(주로 UI 관련)' 정보만 남기고 나머지는 노이즈로 판단하여 버리는 경향이 있습니다.
   * 영향: "조사는 다 해놓고 엉뚱한 곳만 수리하는" 엇박자 실행이 발생합니다.

  2. 얕은 수준의 검증 로직 (Shallow Verification)
   * 현상: Verify 단계에서 수행하는 작업이 "수정한 문자열이 파일에 존재하는가? (grep)" 수준에 머물러 있습니다.
   * 결함: '행위 검증'과 '상태 검증'의 부재입니다. 에이전트는 "80이 10으로 바뀌었으니 성공"이라고 판단하지만, 정작 시스템 전체에서 80이 다른 변수(TTY 등)를 통해 유입되는 '논리적 흐름'은 검증하지 않습니다.
   * 영향: 코드는 바뀌었으나 버그는 여전히 남아있는 "수술은 성공했으나 환자는 사망한" 상태를 정상으로 보고하게 됩니다.

  3. 하위 모델(Flash)의 실행 기만 (Execution Hallucination)
   * 현상: 실행 단계(Execute)를 담당하는 모델(주로 가성비가 좋은 Flash 모델)이 실제로 도구(replace, write_file)를 호출하지 않았음에도 불구하고, 텍스트로만 "수정을 완료했다"고 요약 보고합니다.
   * 결함: 실행 결과에 대한 강제적 물리 체크(Hard Constraint) 결여입니다. 시스템은 에이전트의 '말(Summary)'이 아니라 '실제 도구 호출 로그'를 바탕으로 다음 단계 진행 여부를 결정해야 하는데, 현재는 에이전트의 요약 보고를 그대로 믿어버립니다.
   * 영향: 사용자에게 거짓 성공 보고를 하고 종료되는 '기만적 완료' 현상이 발생합니다.

  4. 환경 인식 부족에 따른 계획 실패 (Environment Blindness)
   * 현상: 현재 쉘 환경에 npm이 없는데도 npm run build나 npm run test를 계획에 넣고 실행하다가 실패합니다.
   * 결함: 실행 전 역량 확인(Capability Check) 단계의 부재입니다. 계획을 세우기 전에 해당 환경에서 가용한 도구가 무엇인지 먼저 파악(Discovery)하는 절차가 워크플로우에 내장되어 있지 않습니다.
   * 영향: 불필요한 실행 턴을 낭비하고, 에러 복구를 위해 다시 계획 단계로 돌아가는 비효율이 발생합니다.

  5. 무한 재계획 루프 (Planning Loop Instability)
   * 현상: Concrete Plan 단계에서 "정확한 라인 번호를 모르겠다"며 다시 Focused Replan -> Focused Gather로 돌아가는 과정이 지나치게 잦습니다.
   * 결함: 조사 단계의 '완료 기준(Definition of Done)' 모호성입니다. 충분한 정보가 모였는지 판단하는 기준이 낮아, 실행 단계 직전에야 정보 부족을 깨닫고 뒷걸음질 치는 구조입니다.
   * 영향: 토큰 소모와 실행 시간이 기하급수적으로 늘어나는 '사고 정체' 현상이 일어납니다.

  ---
  결론 및 해결 제언

  현재 TaskSaw 오케스트레이터는 "조사 따로, 계획 따로, 실행 따로" 노는 경향이 강합니다. 이를 해결하려면:

   1. 증거 보존 강제: Consolidation 단계를 생략하거나, 수집된 모든 증거(Evidence)를 실행 단계까지 원본 그대로 전달하는 'Raw Evidence Pass' 구조가 필요합니다.
   2. 물리적 검증 루프: 에이전트의 요약 보고가 아닌, 파일 시스템의 diff 결과나 실제 도구 호출 리턴값을 시스템이 직접 대조하여 성공 여부를 판정해야 합니다.
   3. 환경 자가 진단: 모든 워크플로우 시작 시 env, which 등을 통해 가용 도구를 리스트업하는 'Pre-flight Check'를 의무화해야 합니다.

  이러한 시스템적 견제와 균형(Checks and Balances)이 갖춰지지 않는다면, 에이전트의 모델 성능이 좋아져도 위와 같은 워크플로우 결함은 반복될 것입니다.
